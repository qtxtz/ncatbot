# 元文档

> 文档编写与维护规范 — 目录编排、内容规范、写作风格、维护流程、AI 阅读优化策略。

本文件供 AI Agent 参考，描述 NcatBot 文档的结构约定、内容规范、写作风格和维护流程。

---

## 一、目录编排

### 1.1 整体结构

```text
docs/
├── README.md          # 文档中心首页：导航表 + 完整目录树 + 详细目录
├── architecture.md    # 架构总览（不限 locs）
├── concepts.md        # 核心概念速查（不限 locs）
├── guide/             # 使用指南（面向用户）
├── reference/         # API 参考（面向开发者）
├── contributing/      # 贡献指南（面向贡献者）
└── meta/              # 文档规范（面向 AI Agent）
    └── README.md      # 本文件
```

### 1.2 README.md 的用途与结构

`docs/README.md`（根索引）是**唯一的详细索引**，包含：

1. 完整目录树（展示所有文件）
2. 各文档的详细描述（几句话介绍内容与用途）
3. 推荐阅读路径

子目录 `README.md` 充当该目录的 **Cheat Sheet 式速查入口**，包含：

1. **一句话定位** — 本目录覆盖什么内容
2. **Quick Start 代码示例** — 包含完整 import 的可运行代码片段，展示最常见的 1–3 个操作场景（插件上下文 + API 调用），让读者快速上手
3. **Quick Reference** — Cheat Sheet 式速查表，以**方法签名表 / 对比表 / 组件表**为主体，覆盖本目录 80% 常用 API 与操作。读者在大多数情况下只需阅读 QR 即可完成工作，子文档是深入补充
4. **本目录索引** — 带简要说明的文件列表
5. **交叉引用**（可选）— 相关目录链接

> **重要**：子目录 README **必须**包含 Quick Start 代码示例。纯表格 README 缺乏上下文，读者无法直接上手。代码示例应涵盖：获取 API 客户端的方式（`self.api.xx`）、最常用的调用模式、必要的 import 语句。参见 `docs/guide/api_usage/README.md` 作为范例。

> 子目录 README 的 Quick Reference 段使用规范化标题 `## Quick Reference`，便于 AI 通过 grep 快速定位。

### 1.3 文件命名约定

| 场景 | 命名格式 | 示例 |
|------|----------|------|
| 有明确阅读顺序的系列文档 | `数字前缀.主题.md` | `1.quick-start.md` |
| 同一主题拆分为上下篇 | `数字+字母.主题.md` | `3a.loading.md` / `3b.unloading.md` |
| 无强制顺序的参考文档 | `主题.md` | `segments.md` |
| 下划线风格（reference/ 惯例） | `数字_主题.md` | `1a_message_api.md` |

> **原则**：同一目录内风格保持一致；不要混用 `-` 与 `_`。

### 1.4 一级目录用途边界

| 目录 | 面向对象 | 内容侧重 |
|------|----------|----------|
| `guide/` | 使用插件的开发者 | 任务导向（"如何做 X"） |
| `reference/` | 查 API / 类型细节 | 参考导向（类、方法、参数列表） |
| `contributing/` | 框架贡献者 | 内部实现、设计决策、开发环境 |
| `architecture.md` | 所有人 | 系统全局视图、模块关系 |
| `concepts.md` | 所有人 | 核心概念速查、术语定义、概念关系 |

## 二、内容规范

### 2.1 规模限制

- **普通文档文件**：目标 < 250 locs，硬限制 ≤ 400 locs（含空行）。
- **超限处理**：
  - 较大主题 → 在同一目录下拆分为 `Xa.主题.md` / `Xb.主题.md`
  - 非常大的主题 → 独立子目录，目录内放 `README.md` + 若干子文件
- **合并优先**：当两个相关文件合并后仍在 400 locs 以内时，优先合并为单文件以减少导航负担。
- **例外**：以下文件豁免 400 locs 限制：
  - `architecture.md` — 架构全景，内容天然庞大
  - `concepts.md` — 核心概念地图，需要完整覆盖所有核心抽象
  - `docs/README.md` — 详细语义索引，每项需要几句话的描述

### 2.2 渐进式披露

- **README.md** → Cheat Sheet 覆盖 80% 常用操作（表格为主，代码为辅）
- **中级文档** → 覆盖常见进阶场景，不展开边缘情况
- **参考文档** → 完整列出所有 API，包括边缘情况和注意事项

不要在快速入门文档里出现复杂配置；不要在参考文档里花大量篇幅写教程。

### 2.3 每类文档要包含的内容

**指南文档（guide/）**
- 明确的**前提条件**（依赖哪些知识/安装）
- 可直接运行的**代码示例**
- 小节标题用动词短语（"注册事件处理器" 而非 "事件处理器"）
- 篇末**延伸阅读**链接（指向参考文档或下一篇指南）

**参考文档（reference/）**
- 类/函数签名（完整类型注解）
- 参数表格（参数名 | 类型 | 默认值 | 说明）
- 至少一个最简使用示例
- 异常/边界行为说明

**贡献文档（contributing/）**
- 内部实现细节，不需要对用户友好
- 可以包含 ADR（架构决策记录）
- 说明为什么这样设计，而不仅是怎么实现

### 2.4 代码块规范

- 始终标注语言标识符：` ```python ` / ` ```toml ` / ` ```bash `
- 示例代码必须**可以直接复制运行**（有完整 import）
- 输出/注释以 `# ...` 行内标注，不单独列出 stdout

## 三、写作风格

### 3.1 语言与语气

- 使用**简体中文**；技术术语（类名、方法名、参数名）保留英文原文。
- 语气：直接、简洁，避免废话（例如"我们将会介绍..."）。
- 句式偏短；一个知识点一个段落。

### 3.2 标题层级

- H1（`#`）：仅文件标题，每文件只有一个
- H2（`##`）：主要章节
- H3（`###`）：子节
- H4（`####`）：谨慎使用，避免超过 4 层嵌套

### 3.3 跨文档引用

- 使用相对路径链接，例如 `[参考文档](../reference/plugin/1_base_class.md)`
- 引用时在括号内附一句说明，例如 `（参见 [ConfigMixin](...)）`
- 禁止硬编码网站绝对路径（文档可能在多处部署）

### 3.4 禁止事项

- 不要在文档中列出 TODO / FIXME / 占位文本
- 不要复制粘贴大段源码
- 不要在一个文件中混合多个不相关主题

## 四、维护规范

### 4.1 新增文档时需同步更新的文件

| 操作 | 需要同步更新的文件 |
|------|-------------------|
| 在某目录下新增 `.md` 文件 | `docs/README.md`（目录树 + 详细索引）；该目录的 `README.md`（链接列表按需添加） |
| 新建子目录 | `docs/README.md` 目录树 + 详细索引 |
| 重命名文件 | 所有引用该文件的文档（用 grep 搜索） |
| 删除文件 | 所有引用该文件的链接（改为链接到替代文档） |

### 4.2 修改已有文档时的原则

- **只修改需要修改的部分**；不做无关的格式调整或重构。
- 若修改导致文件超过 400 locs，先拆分，再修改。
- 修改 API / 行为描述后，检查 `reference/` 和 `guide/` 是否需要同步。

### 4.3 文档与代码同步

- 代码重构时，必须同步更新相关的 `reference/` 文档（签名、参数）。
- 新增公开 API 时，必须在 `reference/` 中添加对应条目，并酌情在 `guide/` 中提供用法示例。
- 删除/废弃 API 时，在原条目添加 `> **已废弃**` 提示，注明替代方案。

### 4.4 验证清单

在提交文档修改之前：

- [ ] 新文件已添加到父目录 `README.md` 的概览表
- [ ] `docs/README.md` 目录树已更新（如新增/删除文件）
- [ ] 所有内部链接可以正确解析（相对路径无断链）
- [ ] 文件 locs ≤ 400（用 `wc -l` 或编辑器行号确认）
- [ ] 代码示例语言标识符已标注
- [ ] H1 标题后紧跟 `> ` 摘要行

## 五、AI 阅读优化策略

本节规定的策略旨在让 AI Agent 更高效地检索和理解文档。

### 5.1 H1 + 摘要行

每个文档文件的 H1 标题后**紧跟一行** `> ` 引用块作为内容摘要。AI 扫描 H1 + 摘要即可判断是否需要深入阅读。

```markdown
# Hook 机制
> 通过 BEFORE_CALL / AFTER_CALL / ON_ERROR 钩子拦截和扩展 Handler 的执行流程。用于权限检查、参数预处理、日志审计等横切关注点。
```

### 5.2 语义化详细目录

`docs/README.md` 的详细目录充当**语义索引**：每项包含关键词密集的描述（几句话），便于 AI 语义搜索命中。

### 5.3 规范化 Quick Reference（Cheat Sheet）

子目录 README 使用统一标题 `## Quick Reference`。AI 可通过 `grep "## Quick Reference"` 快速定位所有快速参考入口。

Quick Reference 的定位：**Cheat Sheet 式速查表**，读者 80% 的情况下只读 QR 就够用，子文档是深入复杂场景的补充。

QR 内容原则：
- **Quick Start 必备**：README 必须在 QR 之前包含 `## Quick Start` 段落，提供带完整 import 的可运行代码示例（插件上下文 + API 调用），展示 1–3 个最常见操作场景
- **表格为主体**：以方法签名表、对比表、组件角色表承载核心信息，按功能分组（每组一个 H3）
- **guide/ QR 策略**（操作导向）：方法/操作表格（方法签名 | 说明），覆盖常用操作和 API 调用方式
- **reference/ QR 策略**（结构导向）：组件角色表（组件 | 职责 | 关键类）+ 常用 API 签名表
- **代码为辅助**：Quick Start 段承载主要代码示例；QR 表格内仅在需要展示调用模式时附少量代码片段
- **完整列举**：常用 API 全部列举（方法名 + 一句话说明），而非只展示几个示例
- 不要为了"完整"而堆砌大段代码

### 5.4 概念地图

`concepts.md` 使用 **术语 → 定义 → 用途 → 关键类 → 参见** 的结构。AI 按术语检索即可理解全局概念并快速跳转到详细文档。

### 5.5 语义化锚文本

文档间引用使用描述性锚文本，例如：
- ✅ `参见 [Predicate DSL 完整 API](../reference/core/2_predicate.md)`
- ❌ `[点击这里](../reference/core/2_predicate.md)`

### 5.6 渐进式深度

三层递进结构，AI 可按需逐层深入：

1. **README / concepts.md** → 快速概览，判断是否相关
2. **guide/** → 任务导向的操作指南
3. **reference/** → 完整 API 签名和参数细节
