# SPEC-07: 迁移计划

> 定义从现有架构到新四层架构的分阶段迁移路线、组件映射和验证标准。

## 1. 组件迁移映射

### 1.1 总览

| 现有组件 | 现位置 | 新位置 | 变化类型 |
|---------|--------|--------|---------|
| MessageRouter | `core/service/builtin/message_router.py` | `adapter/napcat/protocol.py` | **迁移** 核心逻辑到 Adapter |
| PreUploadService | `core/service/builtin/preupload/` | `adapter/napcat/preupload.py` | **迁移** 到 Adapter |
| NapCatWebSocket | `core/adapter/nc/websocket.py` | `adapter/napcat/connection.py` | **迁移** 目录 |
| EventParser | `core/event/parser.py` | `adapter/napcat/event_converter.py` | **迁移** 协议解析到 Adapter |
| BotAPI（具体实现） | `core/api/api.py` + 子模块 | `adapter/napcat/api_impl.py` | **拆分** 接口+实现 |
| IBotAPI Protocol | `core/event/context.py` | `core/api/interface.py` | **升级** Protocol → ABC |
| APIComponent 基类 | `core/api/utils/component.py` | `adapter/napcat/` 内部 | **迁移** 预上传逻辑 |
| UnifiedRegistryService | `core/service/builtin/unified_registry/` | `core/registry/` | **提升** 为核心组件 |

### 1.2 新增组件

| 组件 | 位置 | 说明 |
|------|------|------|
| BaseAdapter | `adapter/base.py` | 适配器抽象基类 |
| NapCatAdapter | `adapter/napcat/adapter.py` | NapCat 适配器主类 |

| RegistryEngine | `core/registry/engine.py` | 注册引擎（原 UnifiedRegistryService） |

## 2. 分阶段实施路线

### Phase 1: 接口定义（无破坏性变更）

**目标**：定义所有抽象接口，不修改现有代码。

**任务**：
- [ ] 创建 `ncatbot/adapter/__init__.py`
- [ ] 创建 `ncatbot/adapter/base.py`（BaseAdapter ABC，按 SPEC-01）
- [ ] 创建 `ncatbot/core/api/interface.py`（IBotAPI ABC，按 SPEC-04）
- [ ] 创建 `ncatbot/adapter/napcat/__init__.py`

**验证**：
- 所有新文件可以正常导入
- 现有功能不受影响

**依赖**：无

---

### Phase 2: Adapter 层抽取

**目标**：将平台相关逻辑从 Core 迁移到 Adapter。

**任务**：
- [ ] 实现 `adapter/napcat/connection.py`（从 `core/adapter/nc/websocket.py` 迁移）
- [ ] 实现 `adapter/napcat/protocol.py`（从 `core/service/builtin/message_router.py` 提取核心逻辑）
- [ ] 实现 `adapter/napcat/event_converter.py`（从 `core/event/parser.py` 迁移）
- [ ] 实现 `adapter/napcat/preupload.py`（从 `core/service/builtin/preupload/` 迁移）
- [ ] 实现 `adapter/napcat/environment.py`（从 `core/adapter/nc/service.py` 迁移）
- [ ] 实现 `adapter/napcat/auth.py`（从 `core/adapter/nc/auth.py` 迁移）
- [ ] 实现 `adapter/napcat/config.py`（从 `core/adapter/nc/config_manager.py` 迁移）
- [ ] 实现 `adapter/napcat/api_impl.py`（NapCatBotAPI，实现 IBotAPI）
- [ ] 实现 `adapter/napcat/adapter.py`（NapCatAdapter，组合上述组件）

**验证**：
- NapCatAdapter 可独立完成：`setup()` → `connect()` → `listen()` → `disconnect()`
- `convert_event()` 正确转换所有事件类型
- `get_api()` 返回的 API 可正常调用所有方法

**依赖**：Phase 1

---

### Phase 3: API 抽象化

**目标**：Core/Plugin 层通过 IBotAPI 而非具体 BotAPI 访问 API。

**任务**：
- [ ] 修改 `core/event/context.py`：`IBotAPI` 类型改为引用 `core/api/interface.py`
- [ ] 修改 Event Mixin：确保 `self.api` 类型为 IBotAPI
- [ ] 修改 `EventDispatcher`：接受 IBotAPI 而非 BotAPI
- [ ] 修改 PluginLoader/BasePlugin：通过 IBotAPI 访问 API

**验证**：
- 通过 IBotAPI 接口调用 API 正常工作
- Event 的 `reply()`, `delete()` 等便捷方法正常工作
- 现有插件代码通过 `event.api.xxx` 调用无需修改

**依赖**：Phase 2

---

### Phase 4: Registry 提升

**验证**：
- 命令注册和匹配正常工作

## 4. 验证标准

| 测试目标 | 说明 |

| IBotAPI 接口合规性 | 验证 NapCatBotAPI 实现所有抽象方法 |

| RegistryEngine 命令匹配 | 命令注册、匹配、执行测试 |
| ServiceManager 拓扑排序 | 依赖顺序和循环依赖检测测试 |

| API 调用 | 通过 IBotAPI 调用群管理、消息发送等 |

## 5. 风险与缓解

| 风险 | 影响 | 缓解措施 |
|------|------|---------|
| Adapter 提取引入回归 | 消息收发功能中断 | 每个 Phase 完成后运行完整测试 |
| 全局注册表 import 顺序问题 | 循环导入、注册丢失 | Phase 4 中仔细处理 import 路径 |
| 第三方插件兼容性 | 插件依赖 `service_manager.unified_registry` | 提供兼容层或迁移指南 |
| EventBus 订阅顺序影响行为 | 处理器执行顺序变化 | 使用 priority 参数控制 |
