# Feat 工作流参考

## 前置：读文档理解设计意图

**先于任何编码工作**，通过 **doc-nav** 技能查阅以下内容：

| 查阅目标 | 路径 | 目的 |
|---------|------|------|
| 架构总览与模块职责 | `guide/11. 架构与概念/1. 架构总览.md` | 确认新功能属于哪个 layer/模块 |
| 现有相似功能的 API 参考 | `reference/<对应层次>/` | 理解现有接口风格与约定 |
| 贡献者内部文档 | `contributing/` | 了解模块内部设计决策 |

**不要在读完相关文档前开始接口设计。**

---

## 接口设计检查点

设计函数/类签名时逐项确认：

- [ ] **命名**：与同模块现有命名风格一致（动词+名词、snake_case 等）
- [ ] **参数**：类型注解完整，可选参数有合理默认值
- [ ] **返回值**：类型注解完整，None vs 异常的选择与现有接口一致
- [ ] **兼容性**：不破坏现有公共接口；跨平台抽象在 `api/traits` 中定义
- [ ] **注册**：新增公共 API 在对应 layer 的 `__init__.py` 中注册

## Docs 委托（**docs-maintenance** 技能）

实现完成后，委托 **docs-maintenance** 技能写文档，提供以下信息：

| 文档类型 | 路径 | 适用场景 |
|---------|------|---------|
| 使用教程 | `guide/` 对应章节 | 新增用户可见功能、新增使用模式 |
| API 参考 | `reference/` 对应层次 | 新增公共接口、事件类型、数据类型 |
| 架构总览更新 | `guide/11. 架构与概念/1. 架构总览.md` | 新增模块或重大结构变更 |
| 贡献者文档 | `contributing/` | 新增内部模块、设计决策说明 |

> Guide 各章节编排说明见 **docs-maintenance** 技能的"文档编排逻辑"一节。
